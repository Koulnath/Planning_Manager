Structure & Flux

L'Architecture du Projet

Notre projet est découpé en couches logiques pour que personne ne se gène lors du développement.

1. Couche Présentation (View)

C'est l'interface graphique. Elle ne connaît rien au SQL ni aux calculs complexes.

  • Utilise Java Swing.
  • Écoute les boutons et affiche les données dans JTable.
  • Lien : com.g9.view
2. Couche Service (Logic)

C'est le cerveau ! Elle fait le pont entre l'UI et la Base de Données.

  • CourseService : Centralise tout le CRUD.
  • ConflictService : Analyse les collisions temporelles.
  • Lien : com.g9.service
3. Couche Persistance (DAO)

C'est l'interface avec MySQL. Elle ne contient que des requêtes SQL.

  • Utilise JDBC via un Singleton.
  • Convertit les ResultSets SQL en objets Java `Planning`.
  • Lien : com.g9.dao

Le Cycle d'une Donnée

L'utilisateur clique sur "Ajouter" → La Vue crée un objet `Planning` → Le Service vérifie s'il y a conflit → Le DAO insère dans MySQL → La Vue rafraîchit le tableau.

Best Practices

Comment coder sans faire d'erreurs ?

Respectez ces 3 lois fondamentales :

Loi 1 : Pas d'Ad-Hoc

N'écrivez pas du code "vite fait". Si vous avez besoin d'une couleur, définissez-la une fois pour toute. Si vous avez besoin d'une date, utilisez un format standard.

Loi 2 : Un fichier = Une responsabilité

Si vous modifiez l'interface, restez dans com.g9.view. Si vous modifiez un calcul, allez dans com.g9.service.

Loi 3 : Documentez en Anglais ou Français clair

Java est en Anglais, mais nos commentaires peuvent être en Français. Soyez clairs.

/**
 * Enregistre un cours s'il n'y a pas de conflit.
 */
public void add(Planning p) { ... }
